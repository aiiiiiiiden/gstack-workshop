## Language Policy

## Problem Statement

The author wants to deeply understand how the Claude Code + gstack harness (skills / hooks / settings / subagents) actually behaves, moment by moment. Surface-level tutorials ("type this command") are saturated; source-code dumps are exhausting. The missing middle is **"on-site dissection"** — how each harness part intervenes during a real build, at the exact moment it intervenes — published in Korean. Primary goal is the author's own learning and skill progression; secondary goal is to publish to Korean developers sitting at the same point.

Deliverables: a Korean blog series under `/docs` + a reference MVP implementation git-tagged at the end of each post.

## What Makes This Cool

- **Harness-as-protagonist Korean series is rare.** There's no shortage of "Claude Code tips" roundups, but pieces that dissect *why skill/hook/subagent look the way they do and when each actually fires* — grounded in a concrete project — are scarce.
- **MVP is not a polished example; it's a dissection stage.** Flutter + Dart server + Postgres + Docker gives the harness many surfaces to latch onto (Dart CLI build, DB migrations, Docker container operation). Pedagogically the most defensible stack for this purpose.
- **Every post ends with code that runs.** Readers can `git checkout v0.N` and land in that exact state. "Read only the post you care about" + "start following along from that post" both work.
- **Learning's failure modes are not hidden.** "I first did it this way, then changed it because..." is a virtue here. The opposite of tutorial-as-finished-answer.

## Constraints

- Deliverable is 8 Korean blog posts + an accompanying reference MVP (git-tag based).
- `/docs/` is the canonical source. Quality must survive being copy-pasted to a blog platform or internal wiki.
- Post length: default 2,000–3,500 Korean characters. Harness-dense posts (P1 environment setup, P4 subagent) may go up to 5,000.
- Harness anatomy section total ≤800 characters/post by default. Exception: **P3 (hook + settings + permission, three mechanical systems) and P6 (hook-limits philosophy) may go up to ≤1,200 characters** since the anatomy is the main event in those posts. The exception compresses other sections (previous-post recap, goal, ending-state) so the overall 5,000-character ceiling still holds.
- Code is counted separately from prose.
- Writing time estimate: 8–16 hours per post at publication grade. 8 posts → writing window 2–4 weeks (separate from actual MVP build time).
- MVP feature scope is fixed: signup, login, thread posts, comments, likes.
- Target reader: intermediate developer. CLI / git / Dart-Flutter fundamentals assumed. Has used or is about to use Claude Code.
- **Version pins**: gstack 0.16.4.0, Claude Code (current installed version), Flutter 3.x, Dart 3.x, Postgres 16, Docker Desktop 4.x. **Main Dart packages must be pinned to exact versions in P1's `pubspec.yaml`** (shelf, shelf_router, postgres, bcrypt, dart_jsonwebtoken, http, etc.). No `^` range operator. `pubspec.lock` must be committed. If the harness API changes mid-series, the affected post states it explicitly; prior posts are not revisited (declared serialization-mode limit).
- **Out of scope** for this series: deep MCP, production security audit, design systems, advanced Flutter state management, CI/CD pipelines.

## Premises (all agreed)

1. The series' heart is harness dissection. MVP is stage. If a reader walks away thinking "Flutter community-app tutorial," the series failed.
2. Readers are intermediate devs, current or aspiring Claude Code users. Not a beginner onboarding.
3. Eight-post linear series: prologue P0 + MVP posts P1–P6 + epilogue P7. Each post has its own TL;DR and takeaway.
4. Every post ends with a runnable git tag. Two deliverables: blog text + tagged reference implementation.
5. Learning journeys and failures are not hidden. The collapse/recovery rhythm is the value.
6. Korean, publication-grade. Terminology: Korean idiom + English in parentheses.
7. Per-post harness mapping (tentative): P0 what-is-harness, P1 first skill invocation, P2 custom skill, P3 hook + permission, P4 subagent basics, P5 plan mode + review, P6 hook deep-dive, P7 mental model.
8. Excluded: deep MCP, production security, design systems, advanced Flutter state management, CI/CD.

## Approaches Considered

### Approach A: Linear Journey — chosen
Every post follows "previous state → this post's goal → harness concept encountered → ending state." The reader sits in the passenger seat. Natural fit for a learning-journal format.

### Approach B: Anatomy + Case Study
Each post has two sections. First: explain the harness mechanism. Second: apply it in the MVP. Structurally safe but textbook-toned.

### Approach C: Incident-Driven — originally recommended, rejected
Open each post with an incident. P5-level execution would be outstanding, but writing burden is high and there's a real risk of fabricating incidents.

## Recommended Approach

**A: Linear Journey.**

Why: It pairs with Builder / learning mode. Lowest writing burden, and the natural journal tone doesn't conflict with publication quality. The harness core doesn't get diluted — each post says "here's a harness concept that came up naturally in this post, and we're dissecting it right here," which keeps Premise 1 (harness is the heart) intact.

Why C was rejected: incident framing is seductive, but applying it consistently across 8 posts means manufacturing incidents in stretches where none exists (P0 prologue, P1 env setup). Minor conflict with the learning-journal tone.

### Post Structure (after applying Linear Journey)

Every post shares this structure:
- **TL;DR** (3–5 lines): which MVP feature this post touches + which harness concept this post dissects
- **Previous Post Recap** (1 paragraph): current project state, v0.N-1 tag reference
- **This Post's Goal** (1 paragraph): what we're building, which harness surface we're drilling into
- **Work Log** (bulk of the body): actual execution, problems encountered, anatomy sections inserted inline
- **Harness Anatomy Sections** (1–3 per post, total ≤800 characters; ≤1,200 for P3/P6): cold, precise descriptions of harness parts touched this post (rendered as boxes / callouts, visually distinct from the narrative)
- **Done When** (3-item checklist): objective criteria that the post is finished. Example: "① `POST /signup` returns 201 with a valid JWT ② Postgres `users` table row confirmed ③ Flutter screen completes signup-login round trip." Pass before tagging.
- **Ending State** (1 paragraph): what the v0.N tag contains, teaser for next post
- **References**: related doc links, files created/modified in this post

### Narrative Honesty Disclosure

"Came up naturally" is the phrasing, but the reality is that **harness topics are pre-assigned per post**. The learning-journal tone is "reconstructing a journey for presentation," not "improvising discovery." For example, P3's `.env` near-miss and P6's `DROP TABLE` moment are either real incidents the author lived through, or **reconstructed scenes** staged during rehearsal. This is disclosed in a footnote on the affected post. Not fabrication, but scripted — and readers get told.

### Fallback When Real Incidents Don't Occur on Schedule

If a planned incident occurs naturally during development, record it as-is. If 14 days pass after the draft of the affected post is started without the incident surfacing, switch to a **pre-scripted scenario** and add a banner at the top of the post: "This post's incident is a reconstruction." Resolves Open Question 3.

- Thread CRUD: Dart server `POST/GET/PUT/DELETE /threads`, pagination.

- Dispatch two subagents in parallel: use the **Agent tool** to invoke two general-purpose subagents in the same message ("dart server implementer", "flutter UI implementer"). No specialized agent types this round — just the Agent tool's default capability.
- Each subagent's context isolation, merging results, integration testing.
- Anatomy: **Agent tool and subagent orchestration** — when to reach for a general-purpose subagent, why self-contained prompts matter, parallel vs. sequential, how subagent results return to the main conversation.
