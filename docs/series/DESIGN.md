## Problem Statement

사용자는 Claude Code + gstack 하네스(스킬/훅/설정/서브에이전트)가 실제로 어떻게 돌아가는지 깊게 이해하고 싶다. 표면적 튜토리얼("이 명령어 치세요")은 이미 충분하고, 반대로 소스 코드 덤프는 지치는 수준이다. 둘 사이에 **"현장 해부"** 층위 — 실제로 한 프로젝트를 짓는 과정에서 하네스 각 부품이 어느 순간 어떻게 개입하는가 — 를 한국어로 공개 출판할 콘텐츠가 비어 있다. 본인의 러닝/레벨업을 주 목적으로 하되, 같은 지점에 있는 한국 개발자에게 공개한다.

산출물: `/docs` 하위 블로그 시리즈 (한국어, 공개 출판 품질) + 각 포스트 끝에서 git tag된 참조 MVP 구현.

## What Makes This Cool

- **하네스가 주인공인 한국어 시리즈는 거의 없다.** "Claude Code 잘 쓰는 법" 팁 모음은 많아도, skill/hook/subagent가 "왜 그렇게 생겼고 언제 뭐가 실행되는지"를 실제 프로젝트 맥락에서 해부하는 글은 희소하다.
- **MVP가 완성도 있는 예제가 아니라 해부 무대다.** Flutter + Dart 서버 + Postgres + Docker 조합은 하네스의 다양한 표면(Dart CLI 빌드, DB 마이그레이션, Docker 컨테이너 운영)에 훅이 걸릴 구멍이 많다. 교재 서비스로 가장 합리적인 선택.
- **매 포스트가 돌아가는 코드로 끝난다.** 독자가 `git checkout v0.N` 하면 그 시점 상태가 재현됨. "읽고 싶은 편만 읽기" + "해당 편부터 따라하기"가 둘 다 가능.
- **학습의 붕괴 과정을 숨기지 않는다.** 이 시리즈에서 "처음엔 이렇게 했다가 왜 이렇게 바꿨다"가 미덕. 정답만 보여주는 튜토리얼의 반대편.

## Constraints

- 산출물은 한국어 블로그 포스트 8편 + 동반 MVP 참조 구현 (git tag 기반).
- `/docs/` 하위가 원본. 블로그/사내 위키에 복사할 때 문제 없는 품질.
- 포스트당 본문 기본 2000~3500자. 하네스 밀도가 높은 편(P1 환경 세팅, P4 subagent)은 최대 5000자까지 허용.
- 하네스 해부 섹션 합계는 편당 ≤800자 기본. 단 **해부가 본문 주역인 P3(훅+설정+권한 3기계)와 P6(훅 한계론)은 ≤1200자까지 허용**. 이 예외는 본문이 다른 섹션(지난 편 요약/목표/끝난 상태)을 압축해서 전체 5000자 상한은 유지.
- 코드는 본문 외로 계산.
- 쓰기 예상 시간: 공개 출판 품질 기준 편당 8~16시간. 8편 = 작성 기간 2~4주(실제 MVP 작업 시간 별도).
- MVP 기능은 고정: 회원가입, 로그인, 스레드 글, 댓글, 좋아요.
- 대상 독자: 중급 개발자. CLI/git/Dart-Flutter 기초는 전제. Claude Code 써봤거나 쓰려는 사람.
- **버전 핀**: gstack 0.16.4.0, Claude Code (현재 설치 버전), Flutter 3.x, Dart 3.x, Postgres 16, Docker Desktop 4.x. **주요 Dart 패키지는 P1의 `pubspec.yaml`에서 정확 버전 고정**(shelf, shelf_router, postgres, bcrypt, dart_jsonwebtoken, http 등). `^` 범위 연산자 금지, `pubspec.lock` 반드시 커밋. 시리즈 도중 하네스 API가 바뀌면 해당 편에서 명시하고 이전 편 재방문하지 않음(연재 방식의 한계로 명시).
- 이번 시리즈에서 다루지 않는 것: MCP 심화, 프로덕션 보안 감사, 디자인 시스템, Flutter 고급 상태관리, CI/CD 파이프라인 구축.

## Premises (모두 합의됨)

1. 시리즈 심장은 하네스 해부. MVP는 무대 장치. "Flutter 커뮤니티 앱 튜토리얼"로 읽히면 실패.
2. 독자는 중급 개발자, Claude Code 경험자/지망생. 초보자 입문서 아님.
3. 8편 선형 시리즈: 프롤로그 P0 + MVP 6편 P1~P6 + 에필로그 P7. 각 편 독립 TL;DR과 테이크어웨이.
4. 매 포스트는 돌아가는 git tag로 끝난다. 산출물 2종: 블로그 텍스트 + 태그된 참조 구현.
5. 학습 여정과 실패는 숨기지 않는다. 붕괴/복구 리듬이 가치.
6. 한국어, 공개 출판 품질. 용어는 한국어 관용 + 괄호에 영문.
7. 포스트별 하네스 매핑 (잠정): P0 하네스란, P1 첫 스킬 호출, P2 커스텀 스킬, P3 훅+권한, P4 subagent 기초, P5 plan mode+review, P6 훅 심화, P7 멘탈 모델.
8. 제외 항목: MCP 심화, 프로덕션 보안, 디자인 시스템, Flutter 상태관리 고급, CI/CD.

## Approaches Considered

### Approach A: 선형 여정 (Linear Journey) — 선택됨
매 포스트가 "지난 편 상태 → 이번 편 목표 → 만난 하네스 개념 → 끝난 상태" 순. 독자는 작업 옆자리에 앉는다. 러닝 저널의 자연스러운 형식.

### Approach B: 해부학 + 임상사례 (Anatomy + Case Study)
포스트가 두 섹션. 앞: 하네스 기제 설명. 뒤: MVP 적용. 구조 안전하나 교과서 톤.

### Approach C: 사건 중심 (Incident-Driven) — 원래 추천했으나 기각
매 포스트를 사건으로 연다. P5 실행은 최강이나 쓰기 부담과 사건 억지 리스크.

## Recommended Approach

**A: 선형 여정**.

채택 이유: 러닝 모드와 짝이 맞다. 쓰기 부담이 가장 낮고 자연스러운 저널 톤이 공개 출판 품질과 충돌하지 않는다. 하네스 심장이 MVP 여정 속에 녹아 있을 뿐 희석되는 것은 아니다 — 각 편이 "이번 편에서 자연스럽게 등장한 하네스 개념을 이 자리에서 해부한다" 구조로 쓰면 P1(하네스 심장)은 지켜진다.

C를 기각한 이유: 사건 프레이밍은 매력적이나 시리즈 전체에 일관되게 적용하려면 사건이 없는 구간(P0 프롤로그, P1 환경 세팅)에서 사건을 억지로 만들어야 한다. 러닝 저널 톤과 소폭 충돌.

### 포스트 구조 (선형 여정 적용 후 확정안)

각 포스트의 공통 형식:
- **TL;DR** (3~5줄): 이번 편에서 건드리는 MVP 기능 + 이번 편에서 해부하는 하네스 개념
- **지난 편 요약** (1문단): 프로젝트 현재 상태, v0.N-1 태그 참조
- **이번 편 목표** (1문단): 이번 편에서 무엇을 만들고, 어떤 하네스 지점을 파고들지
- **작업 기록** (본문 대부분): 실제 진행, 만난 문제, 해부 섹션 삽입
- **하네스 해부 섹션** (1~3개, 합계 ≤800자): 이번 편에서 건드린 하네스 부품을 차갑게 설명 (박스/콜아웃 스타일로 본문과 구분)
- **Done When** (3항 체크리스트): 이번 편이 끝났다는 객관적 기준. 예: "① `POST /signup`이 201과 유효한 JWT를 돌려준다 ② Postgres `users` 테이블에 row 삽입 확인 ③ Flutter 화면에서 가입-로그인 왕복 성공". 태그 찍기 전 통과 조건.
- **이번 편 끝난 상태** (1문단): v0.N 태그가 어떤 상태인지, 다음 편 예고
- **참조**: 관련 문서 링크, 이번 편에서 만든/수정한 파일 목록

### 서사 방식의 솔직한 자기 고지

"자연스럽게 등장"이라고 쓰긴 했지만 실상은 **하네스 주제가 사전에 편별로 배정되어 있다**. 러닝 저널 톤은 "여정을 재구성해 보여주기"이지 "즉흥으로 발견하기"가 아니다. 예컨대 P3의 `.env` 사고나 P6의 `DROP TABLE` 직전 상황은 실제로 겪었거나, 리허설하며 재현한 **재연(reconstructed) 장면**임을 해당 편 하단 주석에 명시. 조작은 아니지만 각본이라는 점을 독자에게 숨기지 않는다.

### 실제 사건이 제때 발생하지 않을 때의 폴백

각 편의 예정 사건이 실제 개발 중 자연 발생하면 그대로 기록. 해당 편 원고 착수 후 14일 내 발생하지 않으면, **사전 스크립트 시나리오**로 전환하고 상단에 "이 편의 사건은 재현 구성입니다"를 명기. Open Question 3 이걸로 해소.

### 포스트별 상세 계획

**P0: 프롤로그 — 하네스란 무엇인가, 왜 이 시리즈를 쓰는가**
- 하네스(harness)의 정의: 코어 LLM 주변의 런타임 컨텍스트 공급자 + 행동 제어자.
- gstack이 하네스의 한 층임을 명시 (Claude Code 위에 얹힌 뼈대).
- 이 시리즈가 다른 Claude Code 글과 어떻게 다른가.
- 프로젝트 전체 계획 (8편 표, 각 편이 무엇을 다루는지).
- 준비물 체크리스트.
- **용어 pin**: 한국어 용어표 정의. "하네스(harness)", "스킬(skill)", "훅(hook)", "서브에이전트(subagent)", "권한 모드(permission mode)", "프리앰블(preamble)", "설정 계층(settings layer)". 이후 8편 전체에서 이 번역만 사용.
- **버전 pin 선언**: Constraints의 버전 고정을 P0 본문에 명시.
- 이 편엔 코드 없음. v0.0 태그 = 이 README와 /docs 초안.
- 하네스 해부 섹션: "하네스 vs Claude Code vs gstack 레이어 구분"
- **Done When**: ① 용어표 확정 ② 8편 일정표 작성 ③ 준비물 체크리스트 재현 가능.

**P1: 환경 세팅 — Flutter/Dart/Postgres/Docker 스택과 gstack 첫 터치** (최대 5000자 허용)
- 프로젝트 레이아웃 결정: `/apps/flutter_client`, `/apps/dart_server`, `/db`, `/docker-compose.yml`, `/docs`.
- Docker compose로 Postgres 로컬 구동.
- Dart 서버 bootstrap (shelf 기반 hello world).
- Flutter 클라이언트 bootstrap (Hello world 화면).
- gstack 설치 및 검증. `/office-hours` 호출은 이 시리즈의 기원으로 **박스/주석으로 간단 언급만** (메인 내러티브는 아님).
- 하네스 해부: **스킬이 로드되는 순간** — 사용자가 슬래시 명령을 치면 내부에서 무슨 일이 일어나는가. SKILL.md가 프롬프트에 주입되는 메커니즘, frontmatter 역할, preamble bash 블록의 의미. 
- v0.1 = 세 컨테이너 hello world + gstack 설치 검증.
- **Done When**: ① `docker-compose up`으로 Postgres 기동 확인 ② `curl localhost:8080/ping` 200 응답 ③ Flutter 앱이 기기/에뮬레이터에 hello world 렌더 ④ **서버와 Flutter 클라이언트 각각의 `pubspec.yaml`이 정확한 버전 핀으로 커밋됨**: 서버 측은 `shelf`, `shelf_router`, `postgres`, `bcrypt`, `dart_jsonwebtoken` 등, 클라이언트 측은 `http`, `provider`(또는 선택한 상태관리), `shared_preferences` 등. `^3.x.x`가 아닌 `3.x.x` 고정. v0.1 태그 시점에 **서버/클라이언트 각각의 `pubspec.lock`**도 함께 커밋.
- **원고 작성 기법**: P1은 실제 글 쓰기 전에 **리허설 bootstrap 1회** 선행(Next Steps 5번). 리허설에서 메모한 내용을 재구성하여 원고 작성. 즉 P1은 "지금 처음 해본 것처럼" 보이지만 실제로는 한 번 거친 경로.

**P2: 회원가입 — 커스텀 스킬로 반복 작업 자동화**
- 회원가입 기능 추가: Dart 서버에 `POST /signup`, password hashing (bcrypt), Postgres users 테이블.
- Flutter에 회원가입 화면 (form + validation + http call).
- 작업 중 반복 감지: DB 마이그레이션 생성 + Dart 모델 + Flutter 모델이 매번 같은 꼴.
- `/scaffold-feature` 커스텀 스킬 작성 (`.claude/skills/scaffold-feature/SKILL.md`).
- 스킬로 재작업: 같은 작업을 스킬 호출 한 번으로.
- 하네스 해부: **스킬 구조 전반** — frontmatter 필드, description의 역할(언제 auto-trigger), SKILL.md 내부에서 Skill tool 호출 불가(재귀) 등 주의사항, skill directory 관례.
- v0.2 = 회원가입 E2E 작동 + 프로젝트용 커스텀 스킬 1개 완비.
- **Done When**: ① `POST /signup`이 201 + JWT 반환 ② Postgres `users` row 삽입 확인 ③ `/scaffold-feature` 호출로 새 기능 하나 더 scaffold 성공(데모).

**P3: 로그인 — 훅으로 비밀 유출 차단, permission model 이해**
- 로그인 기능 추가: JWT 발급, Dart 서버 `POST /login`, Flutter 세션 관리.
- 개발 중 `.env` 파일 생성 (JWT secret, DB 비밀번호).
- **재연 시나리오** (실제 발생하지 않으면 스크립트로 재구성, 상단 고지): Claude Code가 `git add .`으로 `.env`를 스테이징할 뻔한 순간.
- 방어선 구축: PreToolUse 훅 (bash 명령에서 `git add` 시 `.env` 포함 여부 체크).
- settings.json 계층 구조 설명 (user/project/local), local에만 두는 이유.
- permission mode (default/acceptEdits/plan/bypassPermissions) 차이 정리.
- 하네스 해부: **훅 + 설정 + 권한** — PreToolUse/PostToolUse/SessionStart 시점 도해, hook matcher 패턴, settings.json 병합 우선순위.
- v0.3 = 로그인 E2E + `.env` 방어 훅 + settings.local.json 체계화.
- **Done When**: ① `POST /login`이 유효 JWT 반환 + 만료 토큰 거부 ② 훅이 `.env` 포함한 `git add` 차단 (테스트로 검증) ③ settings.local.json이 gitignore됨.

**P4: 스레드 글 — 병렬 subagent로 UI/API 동시 개발** (최대 5000자 허용)
- 스레드 CRUD: Dart 서버 `POST/GET/PUT/DELETE /threads`, 페이지네이션.
- Flutter 스레드 목록 화면 + 상세 화면.
- 문제: UI와 API를 순차로 만들면 시간 낭비. 계약(contract)만 미리 못 박으면 병렬 가능.
- **경량 계약서**: 별도 OpenAPI 도입 대신, `/docs/contracts/threads.md`에 엔드포인트/요청/응답 스키마를 TypeScript 스타일 interface로 서술. OpenAPI는 이번 시리즈에서 도입하지 않음(또 하나 가르칠 것이 늘어나서).
- subagent 2개 병렬 디스패치: **Agent tool**을 써서 general-purpose subagent 2개를 같은 메시지에서 병렬 호출("dart 서버 구현자", "flutter ui 구현자"). 전문 agent 타입 도입은 하지 않음(Agent tool의 기본 기능만 사용).
- 각 subagent의 컨텍스트 분리, 결과 합치기, 통합 테스트.
- 하네스 해부: **Agent tool과 subagent orchestration** — general-purpose subagent의 쓰임, 프롬프트 자립성(self-contained prompt)의 중요성, 병렬 vs 순차, subagent 결과가 메인 대화로 돌아오는 방식.
- v0.4 = 스레드 CRUD E2E + 병렬 subagent 사용 기록.
- **Done When**: ① 4개 엔드포인트 통합 테스트 통과 ② Flutter 목록/상세 화면 렌더 ③ 2개 subagent 병렬 호출 기록이 `/docs/series/04-threads.md` 본문에 재현 가능 ④ **병합 후 검증**: 두 subagent 결과 머지 후 통합 테스트가 재실행되어 스키마 불일치(필드명/타입) 없음 확인. 테스트 레드면 태그 금지.
- **Reader 난이도 주의**: 이 편은 중급 독자에게도 가장 빡셀 것. 도입부에 "이 편은 이전 편들보다 난이도가 있다" 명기.
- **Subagent 분기 리스크**: 각 subagent가 같은 contract를 다르게 해석할 여지가 있음(e.g., `createdAt` vs `created_at`). 본문에 "내가 두 결과를 어떻게 병합했고 무엇이 충돌했는가"를 1절로 배정 — 실제 가치 있는 장면.

**P5: 댓글 — Plan mode와 review 루프로 신중하게 만들기**
- 댓글 기능 추가: 스레드에 1-depth 댓글 (초반은 nested 제외).
- 댓글은 권한 이슈(본인 댓글만 삭제) 때문에 신중 필요 → Plan mode로 진입.
- Plan mode에서 설계 → ExitPlanMode → 구현 → gstack 내장 `/review` 스킬(사내 커스텀 아님)로 검수.
- review 중 발견한 이슈(e.g., N+1 쿼리) 수정.
- 하네스 해부: **Plan mode + review 싸이클** — plan mode 제약 규칙, ExitPlanMode가 실제로 하는 일, gstack `/review` 스킬이 코드 읽을 때 무엇을 기준으로 삼는지.
- v0.5 = 댓글 E2E + 리뷰 이터레이션 1회.
- **Done When**: ① 본인 댓글만 삭제 가능 (타인 댓글 삭제 시도 403) ② Plan mode 기록 캡처가 본문에 인용 ③ `/review` 스킬 이슈 1개 이상 수정 기록.

**P6: 좋아요 — 파괴적 DB 명령 차단 훅, 경쟁조건**
- 좋아요 기능: `POST /threads/:id/like`, `DELETE /threads/:id/like`, 카운터.
- 경쟁조건 이슈 (같은 유저 동시 클릭 → 중복 좋아요): Postgres unique constraint로 방어.
- **재연 시나리오**: `DROP TABLE likes;` 실행 직전. 이걸 차단하는 훅.
- 이번 편은 P3 훅을 확장 — 파괴적 SQL 패턴 매칭. 단 훅은 **`psql` 한 경로**만 방어하며, `docker exec` 경유·ORM 경유 등 우회 경로가 있음을 명시. "완벽 샌드박스가 아니라 자주 하는 실수를 잡는 안전망" 관점.
- Postgres 트랜잭션과 SELECT FOR UPDATE는 **1문단 + 외부 링크**로 제한. 경쟁조건 교과서가 되지 않게.
- 하네스 해부: **훅의 한계와 설계 원칙** — 훅이 완벽한 샌드박스가 아니라는 점, 우회 경로가 존재하고 훅은 '자주 하는 실수'를 잡는 안전망이라는 관점.
- v0.6 = 좋아요 E2E + 파괴적 명령 방어 훅 강화.
- **Done When**: ① 동시 좋아요 2회 요청 시 중복 row 생성 안 됨 ② `psql -c 'DROP TABLE likes;'` 차단 확인 ③ 우회 경로 1개(e.g., docker exec)가 여전히 통과함을 본문에 기록(안전망 관점 강화).

**P7: 에필로그 — 하네스 멘탈 모델, 다음 스텝**

- v0.7 = 완성된 MVP + 완성된 시리즈. (추후 최종 공개 출판 시 v1.0 재태깅 여부는 선택.)
- **Done When**: ① 멘탈 모델 다이어그램 텍스트 완성 ② "건드리지 않은 것" 목록 각각 1줄 소개 + 외부 링크 ③ 독자가 받아갈 '다음 3개 실험' 제시.
