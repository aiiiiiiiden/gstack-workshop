# P7 — 에필로그: 한 장의 멘탈 모델, 그리고 다음 실험

## TL;DR

- 이 편은 새 코드를 만들지 않습니다. v0.6까지 쌓인 여섯 편을 한 장의 멘탈 모델로 압축하고, 시리즈가 의도적으로 비워 둔 영역을 한 줄씩 짚습니다.
- 멘탈 모델 한 장: 사용자 입력 → Claude Code 셸 → (스킬 주입 + 설정 병합 + 권한 모드 + 도구 스키마 로드) → LLM → 도구 호출 → (PreToolUse 훅) → 도구 실행 → (PostToolUse 훅) → 모델 컨텍스트 추가 → 사용자 화면. 서브에이전트는 LLM 단계에서 별도 컨텍스트 창으로 분기되었다가 결과만 메인으로 돌아옵니다.
- 시리즈가 비워 둔 영역: MCP, Memory, ToolSearch, Learnings, community 모드, OpenClaw 호환 — 각 한 줄 + 외부 링크. 그리고 독자가 곧장 시도해 볼 만한 세 가지 실험.

## 지난 편 요약

v0.6에서는 좋아요 기능을 데이터 모델 수준의 멱등성(`(user_id, thread_id)` 복합 PK + `ON CONFLICT DO NOTHING`)으로 봉합하고, `DROP TABLE` 패턴을 잡는 PreToolUse 훅 한 줄을 더했습니다. 또한 그 훅이 통과시키는 경로(`docker exec` 셸, ORM, 변형 SQL)를 같은 본문에 적어 안전망의 폭을 정직하게 그렸습니다. 이를 통해 훅은 "막아 주는 마법"이 아니라 "한정된 안전망"이라는 인식이 시리즈에 박혔습니다.

이번 편은 이 인식 위에 한 장의 그림을 얹습니다. 여섯 편에서 다룬 부품들이 한 번의 사용자 입력을 처리하는 과정에서 어떤 순서로 끼어드는지를 한 장으로 압축하는 것이 이 편의 중심입니다.

## 한 장의 멘탈 모델

이 섹션은 여섯 편에서 다룬 하네스 부품들을 한 번의 사용자 입력 처리 흐름 위에 배치합니다. 이 그림이 이번 편의 해부 박스를 대신합니다. 단일 흐름 위에서 여섯 편의 부품을 교과서 톤으로 배치한 것이기 때문입니다.

```
사용자 입력
   │
   ▼
┌─────────────────────────── Claude Code 셸 ───────────────────────────┐
│  ① 슬래시 명령 해석 ──→ 스킬 매칭 ──→ SKILL.md 본문 주입               │
│  ② 설정 병합: user → project → local (later wins)                    │
│  ③ 권한 모드 결정: default / acceptEdits / plan / bypassPermissions  │
│  ④ 사용 가능한 도구 스키마 로드 (Bash, Edit, Read, Agent, …)          │
│  ⑤ SessionStart 훅 발화 (세션 진입 시 실행)                            │
└──────────────────────────────────┬──────────────────────────────────┘
                                   │ (프롬프트 = 시스템 + 스킬 + 사용자)
                                   ▼
                        ┌─────────────────────┐
                        │       LLM           │
                        │  (Opus / Sonnet)    │
                        └──────────┬──────────┘
                                   │ (도구 호출 결정)
                                   ▼
                          ┌────────────────┐
                          │  PreToolUse 훅  │── block 시 도구 실행 취소
                          └────────┬───────┘    + 사유를 모델 컨텍스트에 추가
                                   │ allow
                                   ▼
                          ┌────────────────┐
                          │   도구 실행     │  (Bash / Edit / Read / Agent …)
                          └────────┬───────┘
                                   │ 결과
                                   ▼
                          ┌────────────────┐
                          │ PostToolUse 훅  │── 결과 가공 / 부수 효과
                          └────────┬───────┘
                                   │
                                   ▼
                       모델 컨텍스트에 결과 추가
                                   │
                                   ▼
                         사용자 화면에 응답 출력

   ▲ 서브에이전트 분기: LLM이 Agent 도구를 호출하면 별도 컨텍스트 창이
     생성되고, 같은 디스크를 보지만 메인의 대화 히스토리는 보지 못합니다.
     종료 시 결과 메시지 한 덩어리만 메인 LLM의 컨텍스트로 돌아옵니다.
```

이 그림이 담는 다섯 가지 핵심은 다음과 같습니다. 첫째, 하네스는 LLM 앞뒤 두 자리에서 동작합니다. 앞에서는 컨텍스트를 빚어 주고, 뒤에서는 도구 호출을 거릅니다. 즉, 훅 없이 하네스를 설명하려면 앞뒤 두 자리를 모두 짚어야 합니다. 둘째, 훅은 도구 호출의 매개에 끼어들지 행위 자체에 끼어들지 않습니다. 셋째, 권한 모드는 어떤 도구가 로드되는지를 바꿉니다. 반면 Plan 모드에서는 Edit/Write/Bash가 통째로 빠집니다. 넷째, 설정 계층은 동일 키에 대해 later-wins로 병합되며, 민감 키(JWT 시크릿 등)는 local 계층에 두어 git에서 떼어 냅니다. 다섯째, 서브에이전트는 LLM 단계의 분기로 표현되고, 메인과는 디스크만 공유합니다.

이 다섯 가지를 외우면 "왜 이 명령은 막혔는지", "왜 저 스킬은 안 떠올랐는지", "왜 서브에이전트가 같은 계약을 다르게 해석했는지" 같은 질문에 그림 위에서 손가락으로 짚으며 답할 수 있습니다. 그림이 없으면 같은 질문에 매번 처음부터 추론해야 합니다.

## 의도적으로 비워 둔 영역

이 섹션은 시리즈가 다루지 않은 영역을 한 줄씩 적습니다. 비움도 설계입니다. 한 시리즈가 모든 것을 다루려 하면 어느 것도 깊지 못합니다. 이 시리즈는 "하네스의 핵심 골격"을 압축하는 데 집중했고, 다음 영역은 다음 학습 주기로 미뤘습니다.

- **MCP(Model Context Protocol)**: 외부 도구·데이터 소스를 표준 프로토콜로 LLM에 연결합니다. 이 시리즈는 도구가 모두 로컬 파일 시스템과 도커 컨테이너 안에서 끝나므로 도입하지 않았습니다. 다음 단계: https://docs.claude.com/en/docs/claude-code/mcp
- **Memory**: 세션 간 정보 보존 메커니즘. 사용자/프로젝트 단위로 사실·선호·피드백을 누적합니다. 이 시리즈는 git 커밋과 `/docs`가 사실상의 메모리 역할을 했기 때문에 별도 도입을 미뤘습니다. 다음 단계: https://docs.claude.com/en/docs/claude-code/memory
- **ToolSearch**: 사용 가능한 도구가 많아질 때 필요한 도구만 동적으로 로드하는 메커니즘. 이 시리즈의 도구 셋은 작아서 정적 로드로 충분했습니다. 다음 단계: https://docs.claude.com/en/docs/claude-code/tool-search
- **Learnings**: 세션 중에 쌓이는 학습 메모를 다음 세션이 참조하도록 하는 gstack의 기제. 시리즈 진행 중 자동으로 쌓였지만 본문에서 해부하지 않았습니다. 다음 단계: https://github.com/garrytan/gstack 의 `learnings/` 폴더
- **gstack community 모드**: 사용 패턴을 익명·집계 형태로 업스트림에 보내 도구 개선을 돕는 옵션. 이 시리즈에서는 내용 흐름을 단순화하기 위해 다루지 않았습니다. 다음 단계: https://github.com/garrytan/gstack (저장소 README의 telemetry 항목)
- **OpenClaw 호환**: gstack의 다중 세션 오케스트레이터 호환성. 자동화된 빌드 파이프라인을 짜는 단계에서 의미가 커지며, 단일 개발자 학습 시리즈인 이 편에서는 비워 두었습니다. 다음 단계: gstack 저장소의 `OPENCLAW.md`

이 여섯 영역은 각각 한 편의 시리즈를 새로 시작할 만한 깊이가 있습니다. 이번 시리즈가 해부한 골격 위에 이 영역들을 얹어 가는 순서가 자연스럽습니다.

## 독자가 곧장 시도할 만한 세 실험

이 섹션은 이 시리즈를 끝낸 독자가 자기 환경에서 곧장 해 볼 만한 실험 세 가지를 제안합니다. 각 실험은 본 시리즈가 만든 v0.6 상태에서 시작 가능하고, 한두 시간 이내에 완결됩니다.

**실험 1 — 자신만의 PreToolUse 훅을 한 줄 추가**: P3·P6에서 만든 두 훅을 모델로 삼아, 자기 작업 흐름에서 가장 자주 저지르는 실수 한 종류를 잡는 훅 한 개를 짜는 것이 출발점입니다. 예: `git push --force` 차단, `rm -rf` 차단, `npm publish` 차단. 정규식은 좁게, 본문에는 우회 경로를 적습니다. 한 시간이면 충분합니다.

**실험 2 — 자기 프로젝트용 `/scaffold-X` 스킬을 한 개 작성**: P2에서 만든 `/scaffold-feature`를 모델로, 자기 도메인의 반복 작업을 한 명령으로 묶는 스킬 한 개를 만듭니다. SKILL.md의 프런트매터(frontmatter) `description`을 잘 적는 것이 핵심입니다. 자동 트리거 정확도가 거기서 결정됩니다. 두 시간 이내면 충분합니다.

**실험 3 — 두 서브에이전트로 작은 기능을 동시에 짜 보기**: P4의 패턴을 빌려, 작은 기능 하나를 "서버 구현"과 "클라이언트 구현"으로 쪼개 같은 어시스턴트 턴 안에서 두 Agent 호출을 띄웁니다. 계약 한 장(`/docs/contracts/X.md`)을 먼저 두는 것이 핵심입니다. 병합 충돌 한두 개를 직접 마주치고 손으로 봉합해 보면 P4 본문의 의미가 손에 잡힙니다.

세 실험의 공통 원칙은 "본문에 흔적을 남기는 것"입니다. 시도와 실패와 수정의 순서를 짧게라도 적어 두면, 다음 학습 주기의 출발점이 됩니다.

## Done When

- [ ] 한 장 멘탈 모델 ASCII 다이어그램이 본문에 포함되어 있고, 다섯 가지 핵심 정리가 그림 아래에 한 단락으로 붙어 있습니다.
- [ ] 의도적으로 비워 둔 영역(MCP, Memory, ToolSearch, Learnings, community 모드, OpenClaw 호환)이 각 한 줄 설명 + 다음 단계 링크로 정리되어 있습니다.
- [ ] 독자가 곧장 시도할 세 실험이 각각 시작 상태(v0.6)·예상 소요·핵심 포인트를 갖춘 형태로 제시되어 있습니다.

## 마치며

이번 편에서는 여섯 편의 부품들을 한 장의 그림으로 압축했고, 의도적으로 비워 둔 여섯 영역을 한 줄씩 적었으며, 독자가 곧장 시도할 만한 세 실험을 제안했습니다. 시리즈의 골격은 여기서 닫힙니다.

이 시리즈의 요지는 "도구의 사용법"이 아닙니다. 요지는 "도구의 한계가 어디에 있고, 그 한계가 왜 의도된 설계인지"를 한 그림으로 들고 있는 것입니다. 한계의 위치를 알면 도구는 무서운 자동화가 아니라 다룰 수 있는 부품이 됩니다. 이에 따라 다음 학습은 새 도구를 더하는 일이 아니라, 이 그림 위에서 어떤 부품을 어떤 순서로 더 배치할지 정하는 일이 됩니다.

이 시리즈를 닫는 자리에서 한 가지를 남깁니다. 다음 도구를 익히기 전에, 이미 익힌 도구들이 어떻게 한 그림으로 묶여 있는지를 한 번 그려 보는 일이 거의 항상 더 큰 이득을 줍니다. 따라서 이 시리즈를 끝낸 시점에서 가장 좋은 다음 행동은 위의 세 실험 중 하나를 그날 안에 시작하는 것입니다. 도구는 점이고, 그림은 선입니다. 점이 많아도 선이 없으면 길이 보이지 않습니다.
