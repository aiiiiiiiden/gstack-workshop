# Voice Guide — 시리즈 블로그 톤 앤 매너

> **Note on language**: 이 문서는 Language Policy 예외로 한국어로 작성. 이유: 한국어 블로그 포스트를 쓸 때 참조하는 문서이므로, 언어를 전환하는 인지 비용을 피한다. 예시와 규칙이 동일 언어에 있어야 쓰는 사람이 바로 적용할 수 있다.

---

## 0. 이 가이드의 출처

저자(aiden)의 기존 글 두 편에서 관찰된 톤을 추출.

- "Design Impeccable로 완성도 높은 웹프론트엔드 디자인하기"
- "Supabase + Supabase MCP를 활용해 React 프로젝트에 Supabase Database 도입하기"

두 글은 **도구 소개 + 실제 적용 + 철학적 마무리**의 3부 구조를 공유한다. 이 시리즈(하네스 해부)는 그 틀에 **"반복 회차 여정"**을 얹는다. 기본 톤은 계승, 구조는 조금 다름.

---

## 1. 한 줄 선언

**정중체 해설조. 객관적 인과로 평가를 대체. 저자가 걸어간 경로를 1인칭으로 재구성하되, 감정은 절제하고 "그래서 이게 뭘 말하나"를 마지막 문단에 남긴다.**

---

## 2. 종결 어미

- **기본**: `~습니다 / ~입니다 / ~합니다` 정중체.
- **허용 확장**: 명사형 종결(`~것`, `~이유`), 의문형(`~할까요?`, `~일까요?`) — 문단 처음 시선 끌기에만.
- **금지**: `~했다 / ~이다` 평서체 혼용, 구어체(`~거든요`, `~잖아요`), 반말.
- **일관성 규칙**: 한 편 안에서 종결 어미 레지스터를 섞지 않는다. 블로그 포스트는 처음부터 끝까지 `~습니다`체.

---

## 3. 문장 길이

| 위치 | 패턴 | 예시 감각 |
|------|------|-----------|
| 도입부 첫 문장 | **짧게 (1~2줄)**. 장면 or 개인 루틴으로 시작. | "출근하면 제일 먼저 하는 일은 …을 둘러보는 일입니다." |
| 도입부 2~4문단 | **중간 길이**. 배경과 문제를 복합문으로 제시. | 근거+결론이 한 문장에 들어가도 괜찮다. |
| 본문 설명 | **복합문 우세**. 주장 + 근거 + 부연을 한 문장에 담는다. | "X는 … 도구입니다. Y가 아닌, Z를 위한 도구입니다." |
| 작업 기록 | **짧아진다**. 명령/결과/관찰이 단문 시퀀스로. | "실행합니다. 결과를 봅니다. 이 시점에서 한 가지 문제가 보입니다." |
| 결론부 | **다시 짧게**. 1문단 요약 + 1문단 확장 + 1문단 통찰. | 문장당 40~80자. |

원칙: **도입과 결론은 숨쉬기 좋게. 본문은 밀도 있게.** 복합문에 지치지 않는 리듬을 만드는 건 도입/결론이다.

---

## 4. 접속사·연결어

**자주 쓰는 것 (권장)**
- 즉 / 또한 / 하지만 / 이를 통해 / 이 때문에 / 따라서 / 이에 따라 / 이러한
- 반면 / 한편 / 다만 / 단 (전환·유보)
- 그 결과 / 결국 / 궁극적으로 (인과 결말)

**가능하면 피하는 것**
- 그리고 (너무 느슨함, 단문 이음에는 허용)
- 그런데 (구어적)
- 근데 / 그치만 (반말 냄새)

**원칙**: 논리 고리가 명시적으로 보여야 한다. 접속사로 시작하는 문장이 편당 5~8회면 자연스러움. 그 이상이면 기계적 느낌, 그 이하면 연결이 약해진다.

---

## 5. 1인칭·시점

- **주어**: `저`(1인칭 단수) 기본. 팀/공동체 맥락이 들어갈 때만 `우리`.
- **2인칭 호격 없음**: `여러분`, `당신`, `~해보세요`는 쓰지 않는다. 명령조도 쓰지 않는다. 대신 **3인칭 일반화**로 거리감 유지.
  - ❌ "여러분도 `docker-compose up`을 실행해보세요."
  - ✅ "이 시점에서 `docker-compose up`을 실행하면 …이 뜹니다."
- **저자의 주관**: `저는 ~라고 생각했습니다` / `~라는 인상을 받았습니다`는 한 편에 2~3회까지만. 남발하면 학습 노트처럼 보인다.

---

## 6. 섹션 전환

소제목 **바로 다음 한 문장**은 "이 섹션이 무엇을 하는지" 요약. 본론은 그 다음.

**템플릿**
```
## 소제목
(1문장 요약) — 이 섹션은 …를 다룹니다. / …를 위해 …가 필요합니다.

(본론 시작)
```

**예시**
> ## 훅으로 `.env` 유출 막기
>
> PreToolUse 훅을 써서 `git add`가 `.env`를 포함할 때 명령을 차단하겠습니다.
>
> 먼저 현재 상태를 확인합니다…

소제목 자체는 짧은 명사구 or 동명사구. `어떻게 X를 Y하는가` 같은 장문 제목은 피한다.

---

## 7. 코드·명령어 소개 패턴

**3단 구조**: 설명 → 코드 → 결과 해설.

**템플릿 1 (명령)**
```
(무엇을 할 건지 1문장).

% <명령어>

(결과를 해석하는 1문장).
```

**템플릿 2 (코드 블록)**
```
(의도 1문장. "X를 Y하기 위해 아래처럼 작성합니다.")

```dart
// code
```

(핵심 부분 해설 1~3문장. 줄 단위로 짚지 않는다 — 핵심만.)
```

**원칙**
- 코드 한 덩어리는 **화면 한 스크롤** 안에. 30줄 넘으면 쪼갠다.
- 코드 다음에 **무슨 일이 일어나는지** 한 문장은 반드시 있어야 한다. 코드로 말 끝내지 않는다.
- 결과 스크린샷은 `/docs/assets/<post>/`에. 설명 문장이 앞뒤로 감싼다.

## 8. 기술 용어 번역 규칙

| 분류 | 규칙 | 예시 |
|------|------|------|
| 제품·라이브러리명 | **원어 그대로** | `Claude Code`, `gstack`, `PostgreSQL`, `shelf`, `Flutter` |
| 첫 등장 개념어 | **한글 음차 + (영문)** | `하네스(harness)`, `훅(hook)`, `서브에이전트(subagent)`, `프리앰블(preamble)` |
| 확립된 한글 용어 | **한글만** | `권한 모드`, `설정 계층`, `병렬 호출` |
| 약어 | **첫 등장 시 풀어서 한 번** | `JWT(JSON Web Token)`, `RLS(Row Level Security)` — 이후 약어만 |
| 코드 내부 토큰 | **코드 폰트** | `PreToolUse`, `SKILL.md`, `settings.local.json` |

시리즈 용어 pin(P0 프롤로그에서 확정):
**하네스 / 스킬 / 훅 / 서브에이전트 / 권한 모드 / 프리앰블 / 설정 계층**

한 번 정한 번역은 8편 내내 고정. 섞지 않는다.

## 9. 평가·감정 표현

- **권장**: 객관적 인과로 평가를 암시.
  - ✅ "이 방식은 … 결과를 만들었습니다."
  - ✅ "이 접근은 … 문제를 남깁니다."
- **절제**: 직접 감정 표현은 편당 **1~2회 이하**로.
  - 허용 형용사: `편리합니다 / 아쉽습니다 / 흥미롭습니다 / 놀라웠습니다`
  - 피할 것: `최고입니다 / 정말 좋습니다 / 감동적입니다` (과장)

**규칙**: 평가를 쓸 때는 **그 다음 문장에 근거**가 와야 한다. "아쉽습니다"로 끝내지 않는다 → "아쉽습니다. 이유는 …"

---

## 10. 결론부 템플릿

**3문단 구조**

1. **회수(Recap)**: "이번 포스팅에서는 … 을 살펴봤습니다." 지나온 동선을 한 번 접음.
2. **확장(Reframe)**: 이 포스트의 발견을 한 층 위의 언어로 다시 말함. 구체 → 추상.
3. **통찰 or 다음 예고(Insight or Next)**: 독자가 가져갈 한 문장 or 다음 포스트 티저.

**예시 골격**
```
이번 포스팅에서는 `.env` 유출을 막기 위한 PreToolUse 훅을 구축하고,
settings.json의 계층 구조를 정리했습니다.

이 과정에서 훅은 단순한 방어선이 아니라 …라는 점을 확인했습니다.
권한 모델과 훅은 분리된 주제처럼 보이지만 실상 …입니다.

다음 포스팅에서는 스레드 기능을 만들며 서브에이전트를 병렬로 운영해봅니다.
여기서의 학습이 어떻게 이어지는지 함께 살펴보겠습니다.
```

**금지**: "이상입니다", "읽어주셔서 감사합니다" 같은 상투적 마무리. 마지막 문장은 **생각의 꼬리**를 남긴다.

## 11. 구조 요소

| 요소 | 쓰는 경우 |
|------|-----------|
| 수평선 `---` | 큰 단락(도입/본문/결론) 구분에만. 소제목 사이엔 안 씀 |
| 인용 블록 `>` | 사용자 요청 문장, 외부 인용, 주의 박스 |
| 표 `\|...\|` | 비교, 옵션 나열, 규칙 목록 |
| 이모지 | **쓰지 않음 (이 시리즈 한정)** — 하네스 해부라는 주제 특성상 절제가 톤을 돕는다. 기존 글에서 쓰던 🙂 🤖 ✅ ⚠️는 이 시리즈에서 제외 |
| 강조 `**굵게**` | 편당 5~10회. 용어 정의, 전환 신호, 주의사항에만 |
| 기울임 `*기울임*` | 최소 사용. 인용 내부의 하위 강조 정도 |
| 체크리스트 `- [ ]` | Done When 섹션 전용. 본문에서는 쓰지 않음 |

**이모지 예외**: 이 시리즈의 기존 저자 톤과 달라지는 의식적 선택. 하네스·훅·권한이라는 냉정한 주제에 이모지는 톤을 흐트러뜨린다. 에필로그(P7) 마지막 한 번만 허용 가능(재량).

## 12. 시리즈 고유 구조 (기존 글과 달라지는 부분)

기존 저자 글은 **도구 소개 한 편**이었지만 이 시리즈는 **8편 연재**다. 그래서 각 편에 다음 두 섹션이 추가된다:

### 지난 편 요약 (도입부 직후)
1문단. "v0.N-1에서는 …까지 진행했습니다. 이번 편에서는 …에서 이어갑니다." 독자가 이 편만 읽어도 맥락이 연결되게.

### 하네스 해부 (본문 중간 or 후반)
콜아웃 박스로 본문과 시각적으로 구분. 이번 편에서 만진 하네스 부품을 **교과서 톤**으로 짧게 해부.

```markdown
> **해부: PreToolUse 훅**
>
> PreToolUse는 도구 호출 **직전** Claude Code 런타임이 발화하는 이벤트입니다.
> matcher 필드로 어떤 도구에 반응할지 지정하고, command 필드에 실행할 셸
> 스크립트를 적습니다. 종료 코드 ≠0이면 도구 호출이 차단됩니다.
```

해부 섹션만 톤이 조금 다르다 — 설명조, 단문 우세, 개인 의견 없음. 본문의 1인칭 저널 톤과 의도적으로 대비.

### Done When 체크리스트 (결론 직전)
3항. 태그 찍기 전 통과 조건.

## 13. Before / After 예시

### 예시 1 — 도입부

**Before (학습 노트 톤)**
> 오늘은 훅에 대해서 공부해봤는데요, 훅은 Claude Code의 기능 중 하나로 도구 호출 전후에 스크립트를 실행할 수 있는 기능이에요. 정말 유용한 것 같아서 정리해봤어요.

**After (시리즈 톤)**
> 로그인 기능을 만들던 중, `.env` 파일이 저장소에 올라갈 뻔한 순간이 있었습니다. Claude Code가 `git add .`을 실행하면서 `.env`를 함께 스테이징하려 했고, 이 순간을 방어하기 위해 PreToolUse 훅을 설치했습니다. 이번 포스팅에서는 그 과정을 살펴봅니다.

### 예시 2 — 평가 문장

**Before (과장)**
> 서브에이전트를 병렬로 쓰니 정말 놀라웠습니다! 속도가 2배나 빨라지고 결과도 완벽했습니다.

**After (절제)**
> 서브에이전트 2개를 병렬로 호출한 결과, 순차 작업 대비 소요 시간이 절반 가까이로 줄었습니다. 단 두 결과를 병합할 때 `createdAt` vs `created_at` 같은 스키마 해석 충돌이 한 번 발생했고, 이를 본문 뒷부분에서 다룹니다.

### 예시 3 — 코드 소개

**Before**
```dart
// 아래 코드
Future<Response> signup(Request req) async { ... }
// 이렇게 하면 됩니다.
```

**After**
> `POST /signup` 핸들러는 JSON 바디를 받아 bcrypt로 패스워드를 해시한 뒤 users 테이블에 삽입합니다.
>
> ```dart
> Future<Response> signup(Request req) async {
>   final body = await req.readAsString();
>   final data = jsonDecode(body) as Map<String, dynamic>;
>   final hash = BCrypt.hashpw(data['password'], BCrypt.gensalt());
>   // ...
> }
> ```
>
> 핵심은 `BCrypt.gensalt()`이 호출될 때마다 새로운 salt를 생성한다는 점입니다. 같은 비밀번호라도 DB에 저장된 해시는 매번 다릅니다.

### 예시 4 — 결론부

**Before**
> 오늘은 훅을 배웠습니다. 정말 유용한 것 같고 앞으로 많이 쓸 것 같네요. 다음 글에서 또 만나요!

**After**
> 이번 포스팅에서는 PreToolUse 훅으로 `.env` 유출을 차단하고, settings.json 계층 구조와 권한 모드를 정리했습니다.
>
> 훅은 완벽한 샌드박스가 아니라 "자주 하는 실수를 잡는 안전망"이라는 관점이 이 편의 중심이었습니다. 보안 경계를 훅 하나에 맡기지 않는다는 설계 원칙은 다음 편에도 이어집니다.
>
> 다음 포스팅에서는 스레드 기능을 만들며 서브에이전트 두 개를 병렬로 운영합니다. 이때 하네스가 제공하는 "격리된 컨텍스트"가 어떻게 동작하는지 살펴보겠습니다.

## 14. 체크리스트 (글 발행 전 통과 조건)

초안 완료 후 다음을 점검:

- [ ] 종결 어미가 처음부터 끝까지 `~습니다`체로 일관된가
- [ ] 문장 길이 리듬이 도입-짧게 / 본문-중간~길게 / 결론-짧게로 굽이치는가
- [ ] 접속사가 편당 5~8회로 자연스러운가
- [ ] "여러분/당신" 같은 2인칭이 없는가
- [ ] 평가·감정 표현이 1~2회 이하이고, 각각 근거 문장이 따라붙는가
- [ ] 코드 블록마다 앞뒤 설명 문장이 있는가
- [ ] 용어 번역이 시리즈 pin과 일치하는가 (하네스/스킬/훅/서브에이전트/...)
- [ ] 결론부 3문단 구조(회수-확장-다음)가 지켜졌는가
- [ ] 이모지가 없는가 (에필로그 예외)
- [ ] 하네스 해부 섹션 톤이 본문과 의도적으로 구분되는가
- [ ] Done When 3항이 객관적으로 검증 가능한가

통과 못 한 항목이 있으면 그 편은 드래프트로 되돌림. 태그 금지.

## 15. 이 가이드를 업데이트하는 방식

- 시리즈 진행 중 새로운 패턴이 유용하다고 판단되면 이 문서에 **append**. 기존 규칙을 덮어쓸 때는 "P3부터 변경: ..."으로 날짜/편수 명기.
- 규칙이 서로 충돌하면 **더 최근 편의 결정이 이긴다**. 단 이전 편을 재작성하지 않는다(append-only 태그 규칙과 동일 철학).
- 에필로그(P7) 직전 이 가이드를 전체 리뷰. 쓰이지 않은 규칙은 **삭제**. 쓰인 패턴이지만 문서화 안 된 건 **추가**.
