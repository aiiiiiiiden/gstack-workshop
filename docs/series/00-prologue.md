# 프롤로그 — 하네스란 무엇인가, 왜 이 시리즈를 쓰는가

## TL;DR

- 이 시리즈는 Claude Code 위에 얹힌 **하네스(harness)**가 실제로 어떻게 동작하는지를, 개발자 커뮤니티 MVP를 직접 만들어보며 해부합니다.
- 스택은 Flutter + Dart 서버(shelf) + PostgreSQL + Docker이고, 편마다 `git tag v0.N`이 걸려 특정 시점의 상태를 재현할 수 있습니다.
- 이번 편은 프롤로그입니다. 코드는 없습니다. 대신 용어 pin과 8편의 지도를 확정합니다.

---

## 들어가며

새로운 도구를 만났을 때 가장 먼저 하는 일은, 어떤 관점으로 그 도구를 이해할지를 정하는 일입니다.

Claude Code를 써본 지는 얼마 되지 않았습니다. 처음 며칠은 슬래시 명령을 외우고 프롬프트를 짧게 치는 데 시간을 썼습니다. 어느 순간부터는 같은 작업을 반복하면서 "왜 이 명령은 이 순간 개입하고, 저 명령은 나중에야 개입하는가"라는 질문이 생겼습니다. 도구 안쪽에서 무엇이 왜 그 타이밍에 실행되는지를 설명해주는 글이 의외로 적다는 점이 이 시리즈의 출발점입니다.

시중에는 "Claude Code 잘 쓰는 법" 류의 팁 모음이 충분히 많습니다. 반대편에는 공식 문서와 소스 코드가 있습니다. 다만 그 사이 "현장 해부" 층 — 실제 프로젝트를 짓는 과정에서 하네스의 각 부품이 어느 순간 어떻게 개입하는지를 1인칭으로 재구성한 글 — 은 비어 있다고 느꼈습니다. 이 시리즈는 그 빈 층에 제가 걸어간 경로를 하나 얹어 두기 위해 쓰입니다.

## 이 시리즈가 다루는 것

이 섹션은 시리즈의 범위, 스택, 편별 지도를 한 번에 보여줍니다.

하네스는 이 시리즈의 주인공이고, 개발자 커뮤니티 MVP는 그 주인공이 움직일 무대입니다. MVP의 기능은 회원가입, 로그인, 스레드 글, 댓글, 좋아요의 다섯 가지로 고정했습니다. 기능을 좁게 잡은 이유는, 각 편이 하네스의 특정 부품 하나에 집중하기 위해서입니다. 따라서 이 시리즈가 "Flutter 커뮤니티 앱 튜토리얼"이라는 인상으로 읽힌다면, 그건 실패한 경우입니다.

편별 지도는 다음과 같습니다.

| 태그  | 이번 편에서 만드는 것         | 이번 편에서 해부하는 하네스 개념     |
|------|------------------------------|-----------------------------------|
| v0.0 | 프롤로그, 레포 골격           | 하네스의 층 구성과 해부 표본 선택      |
| v0.1 | 세 컨테이너 hello world       | 스킬이 로드되는 순간                 |

| v0.3 | 로그인 + `.env` 방어 훅       | 훅 + 설정 계층 + 권한 모드           |

각 편은 독립적인 TL;DR과 Done When 체크리스트를 포함하며, 한 편만 읽어도 하네스 개념 하나는 가져갈 수 있도록 씁니다. 그래서 순서대로 따라가는 독자에게도, 특정 편만 집어서 읽는 독자에게도 쓸모가 있기를 바랍니다.

## 하네스라는 단어를 이 시리즈에서는 이렇게 씁니다

이 섹션은 "하네스"의 정의를 시리즈 안에서 못 박고, 이 시리즈가 그 하네스를 어느 표본 위에서 해부하는지를 밝힙니다.

"하네스"는 일상어에서는 말이나 등산 장비에 걸치는 벨트를 뜻합니다. 소프트웨어 맥락, 특히 최근의 AI 코딩 도구 맥락에서는 다음과 같은 뜻으로 굳어지고 있습니다.

> 코어 LLM 주변에 둘러쳐진 런타임 컨텍스트 공급자이자 행동 제어자. 사용자의 입력을 모델에게 전달하기 전에 가공하고, 모델의 출력을 실행하기 전에 개입하며, 도구 호출의 타이밍과 권한을 규정한다.

이 시리즈의 해부 대상은 이 정의대로의 **하네스 자체**입니다. 해부하려면 손에 잡히는 표본이 하나 필요하고, 이 시리즈가 고른 표본은 Claude Code + gstack 조합입니다. Claude Code는 CLI 층에서 입력 전처리·도구 스키마·권한 모드·훅 트리거를 제공합니다. gstack은 그 위에 얹힌 스킬·훅 묶음의 한 예입니다. 이 조합이 아니어도 같은 해부는 다른 하네스에서 반복 가능합니다. 다만 이 시리즈의 본문과 버전 pin과 재현 스크립트는 이 한 표본에 고정되어 있습니다.

따라서 독자가 이 시리즈에서 가져가야 할 것은 "gstack의 스킬 목록"이 아니라 "하네스가 어느 자리에 끼어드는지를 보는 눈"입니다. gstack은 그 눈을 만들어 주는 발판으로 쓰입니다.

> **해부: 하네스의 층 구성과 해부 표본 선택**
>
> 가장 바깥 층은 운영체제와 셸입니다. 그 위에 AI 코딩 CLI 층이 있고, 이 층이 기본 하네스(입력 전처리, 도구 스키마, 권한 모드, 훅 트리거)를 제공합니다. 그 위에 한 층이 더 얹히는데, 스킬 묶음과 훅 프리셋으로 특정 워크플로우(설계 리뷰, 배포, QA 등)를 일관된 방식으로 돌아가게 조립한 층입니다. 이 시리즈는 CLI 층으로 Claude Code를, 스킬·훅 묶음 층으로 gstack을 표본으로 씁니다. 입력은 `사용자 → 셸 → CLI 층(Claude Code) → 스킬·훅 묶음 층(gstack) → LLM` 순서로 흐릅니다. 이 순서를 머리에 두고 이후 편을 읽으면, 어떤 기제가 어느 층의 것인지 헷갈리지 않습니다. 다른 하네스를 쓰더라도 이 층 구성 자체는 대체로 같습니다.

## 이 시리즈에서 고정해두는 용어 (용어 pin)

이후 8편 내내 아래 번역을 유지합니다. 섞지 않습니다.

- **하네스 (harness)** — 위 정의대로.
- **스킬 (skill)** — `SKILL.md` 파일 하나가 정의하는, 슬래시 명령으로 호출되는 워크플로우 단위.
- **훅 (hook)** — 도구 호출 직전·직후·세션 시작 시점에 실행되는 외부 스크립트. `PreToolUse`, `PostToolUse`, `SessionStart` 이벤트가 있습니다.
- **서브에이전트 (subagent)** — 메인 대화에서 Agent 도구로 분기된 독립 실행 유닛. 각자 컨텍스트가 격리됩니다.
- **권한 모드 (permission mode)** — `default`, `acceptEdits`, `plan`, `bypassPermissions` 네 가지가 있습니다.
- **프리앰블 (preamble)** — 스킬이 로드될 때 가장 먼저 실행되는 bash 블록. 세션 메타데이터를 찍고 업데이트 체크를 수행합니다.
- **설정 계층 (settings layer)** — `user / project / local` 세 층으로 나뉘는 `settings.json` 병합 구조.

## 버전 핀

이 섹션은 시리즈 전체에서 고정할 기술 스택 버전을 선언합니다. 이후 편의 예제가 6개월 뒤에도 재현 가능하려면 이 고정이 필요합니다.

이 시리즈 전체에서 아래 버전을 전제합니다. 이후 편에서 하네스 API가 바뀌더라도, 해당 편에서 명시할 뿐 이전 편을 소급해서 고치지 않습니다.

- 표본 하네스 스킬·훅 묶음 층: gstack 0.16.4.0
- 표본 하네스 CLI 층: Claude Code (독자의 현재 설치 버전)
- Flutter 3.x
- Dart 3.x
- PostgreSQL 16
- Docker Desktop 4.x

P1에서 `pubspec.yaml`의 주요 패키지는 `^3.x.x`가 아닌 `3.x.x`로 고정합니다. `pubspec.lock`도 함께 커밋합니다. 이 원칙 때문에 이 시리즈를 따라가는 독자가 6개월 뒤에도 같은 코드를 돌릴 수 있습니다.

## 준비물 체크리스트

시작하기 전에 다음을 확인합니다.

- [ ] macOS 또는 Linux 환경 (Windows는 WSL2 권장)
- [ ] Docker Desktop 4.x 설치 및 실행 가능
- [ ] Flutter SDK 3.x 설치 (`flutter doctor` 통과)
- [ ] Dart SDK 3.x (Flutter 설치 시 동반 설치됩니다)
- [ ] Git 2.x 이상
- [ ] 터미널에서 `claude` 명령 실행 가능 (Claude Code CLI)
- [ ] 표본 하네스 스킬·훅 묶음 층: gstack 0.16.4.0 이상 설치 (`~/.claude/skills/gstack/VERSION` 확인)
- [ ] 에디터 한 개 (VS Code, JetBrains, Neovim 무관)

따라서 이 중 하나라도 빠지면 P1을 시작하기 전에 해결해두는 편이 시간이 덜 듭니다. Flutter 설치만 30분이 필요할 수도 있고, Docker Desktop은 가상화 설정이 꺼져 있으면 부팅되지 않습니다.

## Done When

이 편이 끝났다고 판단하는 기준은 다음 세 가지입니다.

- [ ] 시리즈 용어 pin 일곱 개(하네스 / 스킬 / 훅 / 서브에이전트 / 권한 모드 / 프리앰블 / 설정 계층)가 본문에 정의되어 있다.
- [ ] 8편 편별 지도(표)가 본문에 들어 있다.
- [ ] 준비물 체크리스트가 독자의 로컬 환경에서 그대로 재현 가능하다.

## 마치며

이번 편에서는 시리즈의 범위, 하네스의 정의, 시리즈 전체에서 쓰일 용어 pin과 버전 pin을 확정했습니다. 준비물 체크리스트까지 맞추면 다음 편으로 바로 넘어갈 수 있는 상태가 됩니다.

이 시리즈는 "Claude Code 잘 쓰는 법"과 "소스 코드를 읽어라" 사이의 빈 층을 메우기 위한 글입니다. 따라서 각 편이 어떤 하네스 부품을 어느 순간 건드리는지를 의식적으로 드러내고, 그 부품이 실제 프로젝트 흐름 속에서 어떻게 개입하는지를 1인칭 기록으로 남깁니다. 용어와 버전을 먼저 고정한 이유도 같습니다. 시간이 지나도 재현 가능한 글이 되려면, 전제가 흔들리지 않아야 합니다.

다음 편 v0.1에서는 Flutter 클라이언트, Dart 서버, PostgreSQL 컨테이너 세 개가 각자 hello world를 찍도록 세팅합니다. 그 과정에서 처음으로 슬래시 명령을 하나 실행해보고, 그 순간 하네스 안쪽에서 어떤 일이 일어나는지를 해부합니다. 도구를 쓰기 전에, 도구가 우리에게 무엇을 해주고 있는지를 보는 것이 이 시리즈의 리듬입니다.
