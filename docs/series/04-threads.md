# P4 — 스레드: 두 서브에이전트를 동시에 띄워 API와 UI를 함께 세우기

> **독자 난이도 고지**: 이 편은 이전 편들보다 밀도가 높습니다. 병렬 서브에이전트 오케스트레이션, 프롬프트 격리, 결과 병합의 세 가지 기제가 한 편에 몰려 있기 때문입니다. 필요하다면 두 번에 나누어 읽는 편이 부담이 적습니다.

## TL;DR

- 이번 편에서 만드는 것: 스레드 CRUD 네 엔드포인트, 커서 기반 페이지네이션, Flutter 목록·상세 화면, 그리고 `/docs/contracts/threads.md`에 담긴 경량 계약.
- 이번 편에서 해부하는 하네스 개념: **Agent(에이전트) 도구와 서브에이전트 오케스트레이션**. 언제 일반 목적 서브에이전트에 손을 뻗어야 하는지, 왜 프롬프트가 자기완결적이어야 하는지, 병렬과 순차의 비용 차이, 또한 결과가 메인 대화로 되돌아오는 경로.
- 이 편 끝나면 Dart 서버의 스레드 CRUD가 통합 테스트를 통과하고, Flutter 앱의 목록·상세 화면이 실제 데이터로 렌더링됩니다. 이 편의 기록 중 하나는 "두 서브에이전트가 같은 계약서를 다르게 해석한 순간"과 그 병합 과정입니다.

> **재연 고지**: 이 편의 "두 서브에이전트 호출 로그"는 실제 세션에서 채취한 출력에 요약과 주석을 덧댄 형태입니다. 토큰 소모를 줄이기 위해 무관한 로그 라인을 잘라냈고, 발견된 `createdAt` vs `created_at` 충돌은 실세션에서 발생한 사건입니다.

## 지난 편 요약

v0.3에서는 로그인을 E2E로 붙이고, `.env` 파일이 실수로 스테이징되는 순간을 PreToolUse 훅으로 차단했습니다. 또한 `settings.json`의 세 층(user / project / local)과 권한 모드 네 가지(default / acceptEdits / plan / bypassPermissions)를 훅과 함께 한 편에 묶어 해부했습니다. 이를 통해 "도구 호출이 실제로 실행되기까지 세 개의 게이트를 지난다"는 구조 인식이 생겼습니다.

이번 편에서는 구조가 하나 더 추가됩니다. 서브에이전트라는 층입니다. 서브에이전트는 권한 모드와 훅이 관여하는 "도구 호출의 타이밍"과는 다른 축에 있습니다. 이 축은 **컨텍스트 격리**입니다. 메인 대화와 다른 대화창이 하나 더 생긴다고 보면 비슷합니다. 다른 점은 그 대화창이 사람이 아니라 또 다른 Claude 인스턴스라는 점입니다.

## 이번 편 목표

스레드 CRUD는 흔한 기능입니다. 이 편이 흔하지 않은 이유는 이 기능을 **순차 구현하지 않는다**는 선택에 있습니다. 전형적인 순차 구현은 "API 먼저, UI 나중"입니다. 이 순서에서 UI 엔지니어는 API가 완성되기를 기다립니다. 이번 편은 이 기다림을 없앱니다. 방법은 단순합니다. 계약을 먼저 못 박고, 두 작업을 같은 턴에 병렬로 분기합니다.

이 선택이 실제로 작동하려면 두 가지 조건이 필요합니다. 하나는 계약이 양쪽 작업자에게 같은 언어로 읽혀야 한다는 점입니다. 다른 하나는 두 작업자가 서로의 컨텍스트를 오염시키지 않아야 한다는 점입니다. 두 조건 모두 서브에이전트가 해결 대상으로 삼는 문제입니다.

## 경량 계약: /docs/contracts/threads.md

이 섹션은 UI와 API 양쪽이 공유할 계약을 먼저 고정합니다. 계약 없이 병렬 작업을 시작하면, 두 결과가 병합되는 순간 스키마가 어긋나 통합 테스트가 전부 빨갛게 떨어집니다.

계약의 형식은 OpenAPI가 아닙니다. 이 시리즈는 OpenAPI를 도입하지 않습니다. 도구가 하나 더 늘면 독자가 배워야 할 것이 하나 더 늘기 때문입니다. 대신 TypeScript 인터페이스 스타일 + 엔드포인트 목록이라는 경량 형식을 씁니다. 파일의 전체 내용은 별도 문서로 커밋되어 있고, 이 편에서는 요지만 발췌합니다.

```ts
// /docs/contracts/threads.md (발췌)
interface Thread {
  id: number;
  authorId: number;
  title: string;
  body: string;
  createdAt: string;   // ISO-8601 UTC
  updatedAt: string;
}

interface ThreadListPage {
  items: Thread[];
  nextCursor: string | null;
}

// POST /threads          → 201 Thread
// GET  /threads?cursor=… → 200 ThreadListPage
// GET  /threads/:id      → 200 Thread | 404
// PUT  /threads/:id      → 200 Thread | 403 | 404
// DELETE /threads/:id    → 204 | 403 | 404
```

계약에는 세 가지 결정이 박혀 있습니다. 첫째, 와이어 포맷은 camelCase입니다. 둘째, 페이지네이션은 커서 기반입니다. 셋째, 권한은 작성자만 편집·삭제할 수 있습니다. 이 세 결정은 해석의 여지를 남기지 않기 위해 계약에 명시되었습니다. 해석의 여지는 병렬 작업자 둘이 서로 다른 방향으로 걸어가게 만드는 가장 큰 원인입니다.

## Agent 도구로 두 서브에이전트 병렬 호출

이 섹션은 Agent 도구를 같은 어시스턴트 턴 안에서 두 번 호출해 병렬 실행을 구성하는 방법을 보여줍니다. 핵심은 Agent 도구를 **같은 메시지 안에서 두 번 호출**하는 것입니다. 같은 메시지 안에서 호출된 Agent 도구들은 병렬로 실행됩니다. 서로 다른 메시지로 나누어 호출하면 순차 실행이 됩니다.

```
[메인 세션 → 사용자 턴]
  계약서 /docs/contracts/threads.md를 참고해 두 서브에이전트를 병렬로 띄우자.
  하나는 Dart 서버 구현, 다른 하나는 Flutter UI 구현을 맡는다.

[모델 → 도구 호출 묶음, 같은 어시스턴트 턴 안]
  Agent(description="Dart 스레드 CRUD 구현",
        subagent_type="general-purpose",
        prompt="<자기완결 프롬프트 A>")
  Agent(description="Flutter 스레드 화면 구현",
        subagent_type="general-purpose",
        prompt="<자기완결 프롬프트 B>")
```

두 서브에이전트에 각각 전달하는 프롬프트는 **자기완결적**이어야 합니다. 서브에이전트는 메인 대화의 히스토리를 자동으로 보지 못합니다. 따라서 프롬프트 안에 필요한 모든 맥락이 들어 있어야 합니다. 이번 편에서는 다음 블록을 양쪽 프롬프트에 공통으로 앞세웠습니다.

```
(공통 헤더)
스택: Dart 3.x / shelf 1.4.1 / Flutter 3.x / Postgres 16.
계약: /docs/contracts/threads.md를 전문 인용해 둘 것.
필드 케이싱: 와이어는 camelCase. DB는 snake_case.
타임스탬프: ISO-8601 UTC 문자열.
```

공통 헤더 뒤에 각자 다른 작업 스펙이 붙습니다. Dart 쪽 서브에이전트에게는 "쿼리, 커서 인코딩, 작성자 체크까지 포함한 네 핸들러와 통합 테스트를 작성해라"라고 주었습니다. Flutter 쪽에는 "목록 화면(무한 스크롤 페이지네이션), 상세 화면, 그리고 API 호출을 담는 `thread_repository.dart`를 작성해라"를 주었습니다. 각 프롬프트 끝에는 "출력은 파일 경로 목록과 핵심 결정 요약"이라는 반환 형식 지시를 넣었습니다. 반환 형식을 지시하지 않으면 서브에이전트가 메인 대화로 얼마나 자세히 보고할지를 스스로 결정하게 됩니다.

## 돌아온 두 결과와 첫 충돌

두 서브에이전트가 각자 작업을 마치고 메인 대화로 돌아왔습니다. Dart 쪽은 `apps/dart_server/`에 네 핸들러와 `test/threads_integration_test.dart`를 남겼습니다. Flutter 쪽은 `apps/flutter_client/lib/features/threads/`에 목록·상세·레포지토리 세 파일을 남겼습니다. 각자의 코드만 보면 정상입니다. 문제는 **이음새**였습니다.

첫 통합 테스트에서 드러난 첫 번째 충돌은 타임스탬프 필드 이름이었습니다. Dart 쪽 서브에이전트는 계약서의 `createdAt`를 그대로 썼지만, 모델 직렬화 과정에서 Postgres 로우의 `created_at`을 그대로 JSON으로 덤프하는 숏컷을 택했습니다. Flutter 쪽은 계약서의 `createdAt`를 그대로 파싱하고 있었습니다. 결과적으로 Dart는 `{ "created_at": "..." }`를 내보냈고, Flutter는 `createdAt`를 찾다가 `null`을 받았습니다.

두 번째 충돌은 커서 필드의 타입이었습니다. Dart 쪽은 커서를 빈 페이지에서 `null`로 돌려줬습니다. Flutter 쪽은 빈 페이지에서 `""` 빈 문자열을 기대하고 있었습니다. 계약서에는 `string | null`이라고 명시되어 있었지만, Flutter 서브에이전트는 이 `null`을 정적 분석 단계에서 마주치지 않기 위해 자기 로컬에서 빈 문자열로 기본값을 두는 선택을 했습니다.

세 번째 충돌은 가장 은밀했습니다. Dart 쪽은 `title`이 빈 문자열일 때 `422`를 냅니다. Flutter 쪽은 빈 제목을 클라이언트 단에서 차단하고 있었지만, 차단된 뒤의 에러 UI가 "네트워크 오류"로 일반화되어 있었습니다. 서버가 보낸 `422`가 클라이언트 화면에서는 네트워크 실패처럼 보였던 것입니다.

## 병합 규칙을 정하고 재실행

이 섹션은 세 충돌에서 도출한 병합 원칙과 재검증 절차를 기록합니다. 세 충돌을 본 뒤 정리한 병합 규칙은 두 줄입니다. "계약이 진실이다. 서브에이전트가 계약과 다르게 동작하면 서브에이전트 쪽을 고친다." Dart 쪽은 직렬화 레이어에 camelCase 변환을 넣어 `{ "createdAt": "..." }`를 내보내도록 수정했습니다. Flutter 쪽은 `null` 커서를 빈 페이지로 해석하도록 고쳤고, `422` 응답을 별도 분기로 처리해 제목 유효성 오류를 폼 필드 아래에 보여주도록 UX를 수정했습니다.

수정 자체는 메인 대화에서 직접 했습니다. 서브에이전트를 다시 띄우지 않은 이유는, 이 수정이 **계약과 구현 사이의 얇은 접합 코드**이기 때문입니다. 서브에이전트의 가치는 "넓은 범위의 코드를 한 번에 생성"에 있지, "세 줄 수정"에 있지 않습니다. 병렬 호출의 비용을 세 줄 수정에 치르면 낭비입니다.

병합 직후 통합 테스트를 다시 돌렸습니다. 네 엔드포인트가 모두 초록색을 냈습니다. Done When ④의 "post-merge validation"이 여기서 채워집니다. 이 재검증이 없으면 서브에이전트 병합 작업은 "합쳐 놓고 깨진 줄 몰랐다"로 끝납니다.

## 해부: Agent 도구와 서브에이전트 오케스트레이션

> **해부: Agent 도구와 서브에이전트**
>
> **일반 목적 서브에이전트의 자리**: Agent 도구의 기본 사용은 `subagent_type="general-purpose"`입니다. 기능을 작업 단위로 분리해 병렬화하는 시나리오에서는 이 타입이 적합합니다.
>
> **자기완결 프롬프트의 이유**: 서브에이전트는 메인 대화의 히스토리·메모리·이전 도구 결과를 자동으로 보지 못합니다. 공유되는 것은 오직 프롬프트와 디스크(파일 시스템)뿐입니다. 따라서 프롬프트는 "이 한 덩어리만 보고도 작업이 가능한가"를 통과해야 합니다. 파일 경로로 간접 참조(`/docs/contracts/threads.md를 읽고...`)가 가능한 이유도 디스크가 공유되기 때문입니다.
>
> **병렬 vs 순차의 분기점**: 같은 어시스턴트 턴 안에서 여러 Agent 도구를 동시 호출하면 병렬 실행됩니다. 다른 턴으로 나누면 순차 실행입니다. 병렬은 속도 이득이 있지만, 두 서브에이전트의 결과가 서로를 참조해야 한다면 순차가 옳습니다. 이번 편의 스레드 CRUD처럼 **계약이 먼저 고정되고 두 구현이 독립 영역에 쓰는 경우**가 병렬의 전형입니다.
>
> **결과 반환 경로**: 서브에이전트는 종료 시 하나의 메시지를 메인 대화로 돌려줍니다. 이 메시지는 모델의 대화 컨텍스트에 추가되지만, 사용자 화면에는 요약 수준만 표시됩니다. 이 때문에 "서브에이전트가 무엇을 만들었는지"를 사용자에게 보이려면 메인 모델이 그 결과를 한 번 더 요약해 출력해야 합니다.

이 네 가지 — 일반 목적 선택, 자기완결 프롬프트, 병렬/순차의 분기점, 결과 반환 경로 — 는 병렬 서브에이전트의 "최소 언어"입니다. 이 언어를 익히면 서브에이전트는 "한 번에 두 작업"이라는 마법이 아니라, **계약 기반 병렬 작업**이라는 설계 패턴으로 보이기 시작합니다.

## Done When

- [ ] `POST/GET/PUT/DELETE /threads` 네 엔드포인트가 `test/threads_integration_test.dart`를 통과합니다. `dart test`의 출력에서 네 개의 테스트 그룹이 모두 초록색입니다.
- [ ] Flutter 앱의 스레드 목록 화면이 커서 페이지네이션으로 렌더링되고, 항목을 탭하면 상세 화면으로 전환됩니다.
- [ ] `docs/series/04-threads.md`에 두 서브에이전트 호출의 턴 구조와 `createdAt` 충돌 장면이 재현 가능한 형태로 남아 있습니다.
- [ ] 두 서브에이전트의 결과를 병합한 뒤 통합 테스트를 재실행해 빨간 테스트가 없음을 확인합니다. 필드명·타입 불일치(`createdAt` vs `created_at` 포함)가 모두 해소된 상태임을 확인한 뒤 진행합니다. 이 단계를 통과하지 못하면 `v0.4` 태그를 달지 않습니다.

## 마치며

이번 편에서는 스레드 CRUD를 두 서브에이전트 병렬 호출로 구현했고, 계약서 한 장을 진실의 원천으로 두어 병합 충돌 세 개를 추적·수정했습니다. 또한 Agent 도구와 서브에이전트의 네 가지 속성 — 일반 목적 선택, 자기완결 프롬프트, 병렬/순차 분기, 결과 반환 경로 — 를 해부했습니다.

이 편의 요지는 "서브에이전트는 빠르다"가 아닙니다. 요지는 "서브에이전트는 계약이 있을 때만 빠르다"입니다. 계약이 없으면 두 서브에이전트는 서로 다른 언어를 쓰는 작업자 둘을 한 방에 집어넣은 상태와 같습니다. 각자 훌륭한 코드를 만들어도, 그 코드는 이음새에서 어긋납니다. 이 편이 계약서 작성을 먼저 하고 병렬 호출을 나중에 한 이유도 같습니다. 속도는 계약의 부산물입니다.

다음 편 v0.5에서는 댓글을 붙입니다. 댓글은 권한 결이 미묘해서(자기 글만 삭제 등) 바로 구현으로 들어가지 않고, Plan 모드로 먼저 설계 합의를 한 다음 구현으로 넘어갑니다. 이에 따라 gstack의 내장 `/review` 스킬을 써서 구현 결과를 이터레이션합니다. Plan 모드의 읽기 전용 성격이 어떻게 "설계 먼저" 규율을 강제하는지, `/review`가 어떤 축으로 코드를 보는지가 그 편의 해부 대상입니다.

## 이 편 끝난 상태

v0.4 태그가 가리키는 상태입니다.
